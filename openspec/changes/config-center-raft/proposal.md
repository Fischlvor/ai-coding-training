## 背景

在应用系统中，配置通常分散在代码、数据库、环境变量或多个配置文件中，容易出现以下问题：

- 不同环境的配置难以统一管理
- 配置更新缺少版本记录，回滚困难
- 多实例服务之间无法及时感知配置变化
- 发布过程缺少灰度能力，风险较高
- 分布式场景下，配置元数据容易出现不一致

本变更拟实现一个轻量级的配置管理中心，并使用 Raft 保证配置元数据在多节点间的一致性。

## 目标

本项目目标是实现一个简化版配置管理中心，支持以下能力：

- 按应用、环境、配置分组管理配置
- 记录配置版本历史，支持发布与回滚
- 支持灰度发布策略
- 支持实时配置推送
- 提供多语言 SDK 的接入能力
- 提供可视化管理后台，用于配置管理、发布管理、回滚管理和集群状态查看
- 使用 Raft 对配置写操作进行一致性复制

## 范围

### 包含

- 应用管理：创建、查询、更新、删除应用
- 环境管理：创建、查询、更新、删除环境
- 配置分组管理：按应用和环境组织配置分组
- 配置项管理：新增、修改、删除、查询配置项
- 配置版本历史：每次发布形成可追踪版本
- 发布与回滚：可切换当前生效版本，也可回退到历史版本
- 灰度发布：支持按比例、按目标实例或按标签发布
- 实时推送：客户端订阅配置变更后可接收通知
- 多语言 SDK：提供统一的访问方式，至少覆盖拉取与订阅流程
- 可视化管理后台：提供 Web 页面用于配置管理、版本查看、发布操作与集群状态展示
- Raft 一致性：配置写入通过 Raft 集群复制并提交

### 不包含

- 完整的企业级权限系统与多租户隔离
- 完整兼容 etcd 的全部协议与 API
- 跨机房强一致复制
- 复杂规则引擎式灰度编排
- 完整服务发现功能
- 分布式事务
- Prometheus 指标接入与 Grafana 仪表盘
- 可视化管理后台之外的复杂前端生态集成

## 技术方案概述

系统采用“业务层 + 管理后台 + Raft 一致性层 + 状态机”的结构：

1. 管理后台或客户端通过 API 提交配置变更请求。
2. 业务层将变更封装为 Raft 命令。
3. Raft 集群完成 leader 选举、日志复制和提交。
4. 提交后的命令由状态机应用到配置数据。
5. 配置变更触发订阅通知，推送给已注册客户端。

Raft 只负责配置写入的一致性，不直接负责灰度规则计算、SDK 逻辑、前端展示逻辑或客户端缓存策略。

## 验收标准

- 能创建、查询、更新和删除应用、环境、配置分组与配置项
- 配置发布后能生成新版本，并可查询历史版本
- 支持将某个历史版本回滚为当前生效版本
- 支持至少一种灰度发布方式
- 管理后台可完成配置查询、发布、回滚与集群状态查看
- 客户端订阅后能收到配置变更通知
- 配置写入在多节点 Raft 集群中保持一致
- 节点重启后能够恢复基本状态
- 至少完成一个语言的 SDK 或统一接入文档
